# CLAUDE.md

Instructions for Claude when working on the **AI-Powered Job Matchmaker & Application Tracker**.

## Project Overview

Full-stack app that helps job seekers aggregate roles from multiple portals, score fit against their resume, generate outreach messages, and track applications end-to-end. Acts as an AI career copilot. Originally scoped from a PRD provided directly during initial build — that source PRD was never checked into the repo, so don't look for a `PRD.md` file; treat this document and the current codebase as the source of truth instead. No database — all persistence is client-side `localStorage`.

## Deployment Model: Self-Hosted, No Multi-User Auth

This is a clone-and-run-yourself project — each user runs their own copy locally (or deploys their own instance) and configures their own API keys. This was a deliberate decision, not an oversight: a full auth layer (signup/login, Gmail SSO, per-user credential storage) was considered and explicitly declined in favor of the simpler model, since the project is meant to be published as a public GitHub repo for anyone to clone. Don't build authentication, multi-tenancy, or per-user credential storage without being asked — the Configuration page (see below) exists precisely so a single self-hosted user can supply their own keys without needing an account system.

## Tech Stack & Layout

npm workspaces monorepo:

```
client/   React 19 + Vite + Tailwind 4 + Lucide React + Motion
server/   Express 5 + tsx (dev) / esbuild (build)
```

- **AI**: `@google/genai` (Gemini + Google Search Grounding), model `gemini-3.5-flash` (`gemini-2.5-flash` was deprecated for new API keys mid-project — if Gemini calls start failing outright rather than timing out, check whether the configured model has been deprecated again before assuming a logic bug)
- **Scraping**: `apify-client` (primary) with Gemini Search Grounding fallback
- **Storage**: Client-side `localStorage` for resume + tracked jobs (Firestore planned for Phase 2, not built). API keys (`GEMINI_API_KEY`/`APIFY_TOKEN`) are also runtime-configurable from the app itself — see Configuration below — and persist server-side to `.env`, not localStorage.

## Running & Verifying

- `npm run dev` — runs both via `concurrently`. Server on :4000, client (Vite) on :5173, `/api/*` proxied to the server in dev.
- `npm run build && npm start` — production build; Express serves the built client statically, no proxy/CORS needed in prod.
- `npm run typecheck -w server` / `npm run typecheck -w client` — run after any change; both must stay clean before considering work done.
- **No test suite exists** — there's no `npm test`. Verification has been done by calling routes directly with `curl` and by driving the real browser with a throwaway Playwright script (installed ad hoc into the scratchpad — not part of the repo, not something to assume is still set up). When changing behavior, verify it actually works end-to-end, not just that it typechecks.

## Environment

`.env` at repo root (gitignored; `.env.example` documents the shape, keep them in sync):
- `GEMINI_API_KEY` — required for all AI features; every AI-backed route degrades to a local heuristic when absent or on any failure/timeout.
- `APIFY_TOKEN` — required for real job scraping; falls back to Gemini Google Search Grounding per-portal when absent or when an actor returns nothing.
- `PORT` — server port, default 4000.

**Gemini free-tier quota is tight in practice** (as low as 20 requests/day observed on the dev key). Expect the fallback path to activate during heavy manual testing — that's the graceful-degradation design working, not a bug. `server/src/env.ts` exposes `hasGemini`/`hasApify` booleans; `/api/health` reports both live.

**Keys are also runtime-mutable, not just `.env`-at-boot.** This project is meant to be cloned and self-hosted (see Deployment Model below), so `client/src/pages/ConfigPage.tsx` (reached via the sidebar's "Configuration" section, route `server/src/routes/config.ts`) lets a user paste their own Gemini/Apify keys directly in the running app. `env.ts` exposes `updateKeys()` (updates in-memory state + rewrites `.env` on disk) and an `onKeysUpdated()` listener so `gemini.ts`/`apify.ts` can rebuild their cached SDK clients without a server restart. If you add a new external API key to the project, wire it through this same runtime-update path, not just a boot-time `process.env` read.

## Architecture: Modules

Keep concerns separated along these lines — this is also roughly the file layout under `server/src/`:

- **A. Resume Parser** (`routes/resume.ts`, `services/resumeExtract.ts`, `services/resumeFallback.ts`) — Extracts text from PDF (`pdf-parse`, note its v2 API is a `PDFParse` class, not the old function-call API)/DOCX (`mammoth`)/plain text, sends to Gemini (heuristic fallback), returns `ParsedResume` JSON plus a generated `careerNarrative` (a readable career-summary paragraph shown in the UI as a "Career Snapshot" card).
- **B. Job Scraping Engine** (`routes/jobs.ts`, `services/apify.ts`, `services/jobNormalize.ts`) — Apify scrapers per portal (LinkedIn, Indeed, Wellfound, Naukri, Flexjobs, Google). If empty/failed, falls back to Gemini Google Search Grounding **per portal**. Always normalizes to the `Job` shape before it leaves the server. Results can be filtered client-side by Posted date, Source (portal), and sorted by match score or date.
- **C. Matchmaker & Scorer** (`routes/match.ts`, `services/matchScoring.ts`) — Implements the Resume↔JD Fit Scoring Agent rubric: six weighted 0-5 dimensions (Skill/Experience Overlap 25%, Domain/Industry 20%, Role/Seniority 15%, Quantified Impact 15%, ATS Keyword Coverage 15%, Ownership/Scope 10%), formula `(Σ dim*weight)*20`, bands Excellent (85+) / Strong (70-84) / Moderate (50-69) / Weak (<50). Two call shapes, both routed through the same deterministic `computeOverallScore()`/`bandFor()` in `matchScoring.ts` so Gemini never supplies a trusted final number: (1) `POST /api/match` — single job, on-demand (Analyze drawer), returns the full evidence-based `report` (What's Good/Bad, Needs Improvement, Skill Gaps, Suggested Improvements with before/after, Do This, Don't Do This). **Gemini-only, no deterministic fallback** for this call specifically — evidence-traced 0-5 scoring with resume-line citations can't be meaningfully approximated by a keyword heuristic, so an unconfigured/failed call returns a `503 { error: "unavailable" }` instead of a fabricated report. (2) `POST /api/match/batch` — attempts **one Gemini call for the whole search result set** (not one per job), returning just the six dimension integers per job (no prose) for list-view score badges/sort; this is what keeps bulk scoring to ~1 call/search against the free-tier quota. Unlike the deep report, batch scoring **does** have a fallback: any job Gemini didn't score (unconfigured, call failed/timed out, quota exhausted, or a malformed response entry) falls back to `estimateDimensionsHeuristic()` — a coarse local keyword/ratio estimate, marked `analysis.estimated: true` so the UI shows it distinctly (dashed ring + `~` prefix on the badge, a callout in the drawer's Score Summary) rather than presenting a guess as the real thing. This makes batch scoring the one deliberate exception to "Gemini-only" for Match Scoring — the reasoning: the free-tier daily quota (as low as 20 requests/day, shared across every AI feature) is trivial to exhaust in real use, and going back to a blank/unscored list every time that happens was judged worse than a clearly-labeled coarse estimate. `client/src/pages/HomePage.tsx` also keeps a session-lifetime score cache (keyed by job id, cleared when the resume changes) so re-searching the same jobs, or a completed deep Analyze report, never re-burns a Gemini call on something already scored this session — `JobDetailDrawer.tsx` similarly skips re-fetching the deep report if `job.analysis.report` already exists from a prior open.
- **D. Outreach Generator** (`routes/outreach.ts`, `services/outreachFallback.ts`) — Produces a LinkedIn InMail-style message (~80-150 words) and a cold email (~120-180 words, subject required and must name the exact role), both built from a fixed five-part structure — Hook (exact job title + real link, never fabricated) → Bridge (≤2 concrete achievements) → The Ask (varies: "happy to apply, introducing myself first" if the referrer is a job-poster-type title, otherwise a referral/pointer ask) → Honest Gap (only if a real gap exists) → Low-friction close. Tone stays direct/honest/low-pressure, never corporate-speak (no "align with your goals", "translate into impact", or buzzwords like "visionary"/"passionate"/"guru"/"rockstar"/"ninja" — prefer commas and short sentences over stacked hyphens/semicolons, while keeping the established em-dash Hook style). Auto-generates a plausible referrer contact per job (`synthesizeReferrer` / `isJobPoster` in `outreachFallback.ts` classify the referrer's title to pick the Ask framing). Each draft (LinkedIn message and email independently) has a **Regenerate** action in the UI (`JobDetailDrawer.tsx`'s `RegenerateButton`, `routes/outreach.ts`'s `/regenerate` endpoint) that must produce a genuinely different version, not a reword — when Gemini is configured, the prompt is given the previous draft and told explicitly to vary the opening/leading achievement/ask phrasing; when it isn't (or the call fails), `outreachFallback.ts` rotates through a small set of equivalent phrasings keyed by a `variant` counter the client increments per click, so even the no-AI path produces a different result each time instead of silently returning identical text. The Bridge step (`buildBridge()`/`pickHighlights()` in `outreachFallback.ts`) metric-prioritizes resume achievements — highlights containing a digit sort first, since a quantified bullet ("reduced churn 18%") reads stronger than an unquantified one — and closes with `relevanceClause()`, which names an actual overlapping skill/domain pulled from `job.requirements` (falling back to the company name, then a generic line) instead of canned filler like "maps closely to what this role is asking for". The Bridge is also channel-differentiated: email gets the relevance clause as its own full sentence, the LinkedIn message folds it into one tight sentence. The Honest Gap step guards against a double-noun grammar bug (e.g. "PCI-DSS compliance knowledge experience") by checking whether the gap phrase already ends in a noun like "experience"/"knowledge"/"skills" before appending "experience". Ask/Close phrasing pools were widened from 2 to 4 variants each to reduce repetition across regenerates. `routes/outreach.ts`'s Gemini prompt mirrors all of the above (metric preference, real-skill relevance naming, channel-specific Bridge length, grammar guidance, buzzword ban) — keep the prompt and `outreachFallback.ts` in sync per the rule below.
- **E. Application Tracker** (`client/src/pages/TrackerPage.tsx`, `client/src/components/Kanban*`) — Stages: `Interested` → `Applied` → `Interviewing` → `Offer Received` → `Archived`. Supports manual entries (`ManualJobModal.tsx`) with native HTML5 drag-and-drop between columns. The manual-entry source dropdown includes an **"Other"** option that reveals a free-text field — this is why `Job.portal` is typed `Portal | string` rather than the strict `Portal` union: manually-tracked jobs can carry an arbitrary source name, while scraped jobs (Find Jobs, LinkedIn Job Posts) always still use a real `Portal` value. The standalone `Portal` type itself stays a strict union everywhere it's used for the actual scraping/actor pipeline (`services/apify.ts`'s `ACTORS` map, `jobNormalize.ts`'s per-portal mappers) — only the `Job.portal` *field* was widened.
- **F. Guidance Layer** (built into `services/matchScoring.ts` → `buildGuidance()`) — The Resume↔JD Fit Scoring rubric itself has no gapType/path concept, so this layer derives one deterministically from the six dimension scores/band (not from the rubric doc) to keep this UI feature alive: a Gap Type classification (`hard_filter` | `positioning` | `volume_competition` | `structural_mismatch`), a recommended Path (`apply_standard` | `apply_referral` | `referral_first` | `reframe_then_apply` | `skip`), and Verdict/Why/Do-This/Confidence-note text. `hard_filter` fires when `domainIndustryMatch` or `roleSeniorityMatch` is ≤1; `structural_mismatch` fires on a Weak band, or a Moderate band with both Skill/Experience Overlap and ATS Keyword Coverage ≤2, or (once the deep report has run) ≥4 Skill Gaps outside an Excellent band; otherwise `volume_competition` on high applicant volume/large-enterprise size, else `positioning`. `doThis` reuses the deep report's own `doThis` once one exists; before that (batch/list-view only), a smaller path-keyed template covers it — `gapType`/`path` themselves are always deterministic, never Gemini-influenced. A batch-level "Today's Priority" banner (`client/src/lib/priority.ts`, `TodaysPriorityBanner.tsx`) surfaces an **aggregate insight** across the whole batch — e.g. "3 jobs in this batch look like a strong match (Strong band or better) — 2 ready to apply directly, 1 best approached via referral first" — with 1-2 specific top picks shown as a secondary "Top picks" reference below the insight line, not as the headline. This was deliberately redesigned away from an earlier version that led with named job pills (read too much like "go apply to this specific job") — keep the insight sentence as the primary content; the picks are supporting detail. **Find Jobs only** — the banner was removed from the LinkedIn Job Posts tab entirely (see Module H / "LinkedIn Job Posts Details") once that tab stopped computing match scores at all, since the insight sentence has nothing to summarize without one.
- **G. Configuration** (`client/src/pages/ConfigPage.tsx`, `server/src/routes/config.ts`) — Lets a self-hosted user paste their own `GEMINI_API_KEY`/`APIFY_TOKEN` into the running app rather than editing `.env` by hand. See the runtime-mutable keys note under Environment above.
- **H. LinkedIn Job Posts** (`routes/linkedInPosts.ts`, `services/linkedInPostsApify.ts`, `services/linkedInPostFallback.ts`, `services/linkedInPostNormalize.ts`; client: `LinkedInPostsSearchPanel.tsx`, `LinkedInPostFilterBar.tsx`, `LinkedInPostCard.tsx`, `LinkedInPostCardGrid.tsx`, `state/useLinkedInPosts.ts`) — A second tab inside the Search page (`HomePage.tsx`, alongside "Find Jobs") that finds LinkedIn **posts** announcing hiring (people/companies posting "we're hiring X" — not LinkedIn's structured job board, which is Module B's `LinkedIn` portal instead) via the Apify actor `datadoping/linkedin-posts-search-scraper`. See "LinkedIn Job Posts Details" below for the extraction/classification pipeline. Cards show a cleaned summary paragraph of the post (emoji/hashtags stripped, truncated on a word boundary — `summarize()` in `LinkedInPostCard.tsx`) plus up to 6 requirement tags, and **only** View Post + Track actions — deliberately no Analyze button/drawer for this tab (post cards already show enough at a glance; the Overview/Analysis/Outreach drawer is Find-Jobs-only). This tab intentionally shows no match score, no "verified job posting"/"AI-matched" source badge, no Score filter/sort, and no Today's Priority banner — all removed by deliberate decision (see "LinkedIn Job Posts Details" below), not an oversight; don't re-add any of these without being asked. Role chips and location default to all Product-Management-track titles from `PRODUCT_MANAGEMENT_ROLES` (`searchSuggestions.ts`) except Associate Product Manager, and `"Bengaluru, Karnataka"` respectively — the same `PRODUCT_MANAGEMENT_ROLES` list is also Find Jobs' default title set now (`useJobSearch.ts`'s `DEFAULT_ROLE_TITLES`), replacing an earlier narrower 6-title curated default.

## Data Models (source of truth)

Defined in both `server/src/types.ts` and `client/src/lib/types.ts` — **keep them in sync manually**, there's no shared package. Current shape (has grown since the original scoping conversation — treat the code as the source of truth):

```ts
ParsedResume { name, email, phone?, summary, skills[], experience[{role, company, duration, highlights[]}], education[], noticePeriodMonths?, careerNarrative? }
Job { id, title, company, location, type: "Remote"|"Hybrid"|"In Office", portal: Portal | string, url, description, requirements[], postedAt, referrer?, analysis?, applicantCount?, companySize? }
DimensionScores { skillExperienceOverlap, domainIndustryMatch, roleSeniorityMatch, quantifiedImpactStrength, atsKeywordCoverage, ownershipScopeMatch } // each 0-5
JobReport { verdict, whatsGood[], whatsBad[], needsImprovement[], skillGaps[], suggestedImprovements[], doThis[], dontDoThis[] } // present only after the on-demand deep report succeeds
JobGuidance { verdict, why[], gapType: GapType, path: GuidancePath, doThis[], confidenceNote }
JobAnalysis { matchScore, band: ScoreBand, dimensions: DimensionScores, guidance: JobGuidance, report?: JobReport, customMessage, customEmail }
LinkedInJob extends Job { author: { name, headline?, profileUrl? }, engagement?: { reactions?, comments?, reposts? }, postSource: "structured" | "classified" }
```

`Portal = "LinkedIn"|"Indeed"|"Wellfound"|"Naukri"|"Flexjobs"|"Google"` stays a strict union everywhere it's used for the actual scraping/actor pipeline. Only the `Job.portal` *field* is widened to `Portal | string`, specifically so the Application Tracker's manual-entry "Other" source option can carry an arbitrary free-typed source name — see Module E. `LinkedInJob` is a superset used only by the LinkedIn Job Posts feature (Module H); every function that only needs the `Job` subset (scoring, the shared `JobDetailDrawer`, etc.) works on it unmodified since extra properties on a superset are invisible to and harmless for code that only reads the `Job` fields.

`GapType`, `GuidancePath`, `DimensionKey`, `ScoreBand` (and the `JobGuidance`/`JobReport`/`JobAnalysis` interfaces) live in `server/src/types.ts`, not in `matchScoring.ts` — this was deliberate, to avoid a circular import between the scoring service and the shared types file. Client-only label/style maps for rendering (`GAP_TYPE_LABELS`, `PATH_LABELS`, `DIMENSION_LABELS`, `DIMENSION_WEIGHTS`, `BAND_LABELS`, `BAND_STYLES`) live in `client/src/lib/types.ts` alongside the mirrored types, since the server has no use for display labels.

## Core Pattern: Gemini + Deterministic Fallback, Everywhere (Match Scoring's deep report is the one exception)

Every AI-backed capability has two implementations behind the same interface, with the Gemini path wrapped in try/catch that falls through silently on any error/timeout — this is a hard requirement, not optional polish:

| Capability | Gemini path | Fallback path |
|---|---|---|
| Resume parsing | `routes/resume.ts` → `services/gemini.ts` | `services/resumeFallback.ts` |
| Match scoring — deep report (`POST /api/match`) | `routes/match.ts` → `services/matchScoring.ts` | **none** — Gemini-only by deliberate decision, see Module C |
| Match scoring — batch/list-view (`POST /api/match/batch`) | `routes/match.ts` → `services/matchScoring.ts` | `matchScoring.ts`'s `estimateDimensionsHeuristic()`, result marked `estimated: true` |
| Outreach drafting (initial + regenerate) | `routes/outreach.ts` | `services/outreachFallback.ts` |
| Job search (per portal, if Apify empty/fails) | `services/gemini.ts` (`searchGroundedJobs`) | — (returns empty for that portal, not a crash) |
| LinkedIn post classification (Layer 2, if no structured job card) | `routes/linkedInPosts.ts` (batched `generateStructured()` call) | `linkedInPostFallback.ts`'s `extractHeuristicFields` |

**The deep report specifically** breaks the "always have a fallback" rule: evidence-traced 0-5 dimension scoring with resume-line citations can't be meaningfully approximated by a keyword heuristic, so there is no fallback for `POST /api/match` — an unconfigured/failed call returns `503 { error: "unavailable", reason: "gemini_not_configured" | "gemini_failed" }`, and the client shows a clear unavailable state (Retry button in the drawer), never a fabricated report. **Batch/list-view scoring does have a fallback**, added after real usage showed the free-tier daily quota (as low as 20 requests/day, shared across every AI feature) gets exhausted easily enough that "no fallback" meant entire searches showing zero scores — see Module C for the estimator and the `estimated` flag it sets. What's preserved from the old architecture either way: the formula/band/gapType/path computation (`computeOverallScore()`, `bandFor()`, `classifyGapType()`, `selectPath()` in `matchScoring.ts`) is always deterministic TypeScript — Gemini (or the heuristic estimator) supplies only the six 0-5 dimension integers as data, never a trusted final score. The "one call per job would burn quota" problem the old heuristic solved is now solved differently for the batch path: `POST /api/match/batch` batches an entire search's jobs into a **single** Gemini call (mirroring `routes/linkedInPosts.ts`'s batching pattern) rather than looping one call per job — see Module C. On top of that, `HomePage.tsx`'s session-lifetime score cache and `JobDetailDrawer.tsx`'s skip-refetch-if-report-exists check both exist purely to avoid spending quota re-scoring something already scored this session — check those before adding new Gemini call sites to match scoring.

`generateStructured()` in `services/gemini.ts` takes an optional `timeoutMs` (default 15s). Pass a longer one explicitly for calls generating a lot of output — e.g. resume parsing + career narrative in one call needs 30s (see `routes/resume.ts`), the single-job deep fit report needs 35s, and the batched list-view scoring call needs 40-45s (see `routes/match.ts`). **A silent, unexplained fallback/failure is often actually a timeout, not a real failure** — if output looks wrong or a job comes back unscored, don't assume it's a logic bug before checking whether the Gemini call is simply timing out (call the SDK directly with a small debug script to confirm).

## Job Scraping Details

`services/apify.ts` maps each portal to one specific Apify actor. Each actor has a genuinely different input schema and output shape — when adding/changing a portal integration, verify the actor's real schema via the Apify API (`GET /v2/acts/{id}` for `exampleRunInput`, `GET /v2/acts/{id}/builds/{buildId}` for the full `actorDefinition.input` schema and README) rather than guessing field names. `services/jobNormalize.ts` has one hand-written mapper per portal into the shared `Job` type; wrap each mapper's per-item processing in try/catch so one malformed record doesn't drop the whole batch.

`routes/jobs.ts` fans a search out across every (title × portal) combination (multi-role search is supported via chip input), applies a title-based relevance filter (checking JD text alone is too noisy — e.g. almost any tech job description mentions "product managers" as a stakeholder), dedupes by job ID, and falls back to Gemini grounding per-portal when Apify returns nothing for that portal.

## LinkedIn Job Posts Details

This is a fundamentally different actor/data paradigm than Module B — `datadoping/linkedin-posts-search-scraper` searches LinkedIn **posts** by free-text keyword (input: `keywords: string[]`, `max_posts`, `sort_by`, `date_filter: "past-24h"|"past-week"|"past-month"` — no location parameter at all, and no arbitrary day-granularity beyond those three buckets), returning raw social-post data (text, author, hashtags, engagement, an optional `content` object), not structured job listings. Deliberately kept in its own service files (`linkedInPostsApify.ts`/`linkedInPostFallback.ts`/`linkedInPostNormalize.ts`) rather than folded into `services/apify.ts`'s `ACTORS` map, since that map is keyed by the strict per-portal `Portal` union and this actor doesn't fit that shape.

**Pipeline, in order** (`routes/linkedInPosts.ts`): fetch (requesting `date_filter: "past-month"`, the broadest ≥15-day option available, since there's no native 15-day setting) → hard age-exclude anything over 15 days old (cheapest check, runs first, both to cap payload size and to avoid spending a Gemini call classifying a post about to be discarded anyway) → classify/extract → soft location filter → normalize → return.

**Classification pipeline (structured job-card posts are now excluded, not fast-pathed):**
`extractStructuredJobCard` (in `linkedInPostFallback.ts`) still detects posts where LinkedIn itself attached a structured job card (a `content` object with `content.type === "job"`, confirmed via a live trial run against the actor). It used to qualify those posts as a high-confidence fast path (`postSource: "structured"`). That was inverted by a later decision: this tab is meant to surface **organic** posts only ("someone posted about hiring"), not posts that already look like a job-board listing — Find Jobs already covers structured listings. So in `routes/linkedInPosts.ts`, any post where `extractStructuredJobCard(post)` returns non-null is now **excluded outright**, before classification even runs. Every post that survives to display goes through the remaining classified path, so `postSource` is `"classified"` in practice for every result — the `"structured"` value stays in the type union (harmless looseness, avoids type churn, keeps the door open if this is revisited) but is currently unreachable.

The classified path itself is unchanged: a deterministic Layer-1 heuristic (`passesHiringIntentHeuristic`) requires genuine hiring-intent phrasing (hashtags like `#hiring`, phrases like "we're looking for" — deliberately excludes `#opentowork`, which signals a job *seeker*, not a poster) combined with a role-term match, before anything is shown at all. If Gemini is configured, Layer-1 survivors are batched (not one call per post — controls latency/quota) into a `generateStructured()` call in `routes/linkedInPosts.ts` that must extract title/company/location **verbatim** from the post text or return null; a belt-and-suspenders check additionally rejects any extracted company that doesn't literally appear in the source text. If Gemini isn't configured (or a batch fails), the deterministic fallback (`extractHeuristicFields`) is honest about what it doesn't know: `title` falls back to the searched role (not an invented specific title), and `company` is parsed from the author's LinkedIn headline via an "X @ Company"/"X at Company" pattern, falling back to the generic placeholder `"Hiring company"` — **never** the post author's own personal name (an earlier version did this, and it produced nonsensical outreach like "excited about what Billy Nixon is building," where Billy Nixon was the poster, not the employer).

**Location filtering is soft/best-effort, not authoritative** — the actor has no location field, so `matchesLocationHint` does case-insensitive token matching (e.g. "Bengaluru, Karnataka" → `["bengaluru","karnataka"]`) against the post text/author headline/any structured-card location. The UI says this explicitly next to the filter; don't let it silently imply the same precision as Find Jobs' structured location field.

**Scoring/match-score display was deliberately removed from this tab** (no per-post score badge, no "verified job posting"/"AI-matched" source badge, no Score filter/sort option, no Today's Priority banner) — since job-card posts are now excluded and organic posts are inherently thin on structured signal, a per-post score was judged more noise than signal for this tab specifically. Concretely: `HomePage.tsx`'s `handleLinkedInResults` no longer calls `batchMatch()` for LinkedIn Job Posts results (Find Jobs' `handleResults` still does, unchanged); `LinkedInPostCard.tsx` has no score badge/column; `linkedInPostFilters.ts`'s `SortOption` is date-only (`"date-desc" | "date-asc"`, no score variants); `LinkedInPostFilterBar.tsx` shows only Posted + Sort controls. Note: the old scoring system had a small `ScoringSignals.isLinkedInPost` differentiation kept dormant specifically for this feature (in the now-retired `scoringSignals.ts`) — that hook was removed along with the rest of the old heuristic model when Matchmaker & Scorer was rewritten around the Resume↔JD Fit Scoring rubric (see Module C); since this tab already doesn't call `batchMatch()` at all, nothing about this tab's behavior actually changed. **Find Jobs (any portal) keeps its score badge, Score sort, and Today's Priority banner** — none of that changed.

**Known constraints, surfaced in the UI, not engineered around** (same philosophy as the Gemini quota callout above): free-tier Apify accounts cap at 4 keywords per run — if more than 4 role titles are selected, not all get searched in one run. No per-user usage cap exists on either Apify or Gemini calls for this feature. Both are stated plainly in the search panel's helper text rather than silently limited.

## Client State — Two Patterns, Don't Mix Them Up

- **Persisted across browser sessions** (survives refresh, via `localStorage`): resume + parsed data (`state/useResume.ts`), tracked jobs (`state/useTracker.ts`).
- **Persisted across in-app page navigation only** (in-memory, NOT localStorage): job search results + search form criteria (`state/useJobSearch.ts`), and separately LinkedIn Job Posts results + form criteria (`state/useLinkedInPosts.ts`, distinct property names — `linkedInForm`/`linkedInPosts` rather than `form`/`jobs` — so merging both hooks into the flat `AppContext` doesn't collide). This exists because `App.tsx` conditionally renders `<HomePage />` vs `<TrackerPage />` and unmounts the inactive one — any state that needs to survive a Search↔Tracker switch (or, now, a Find Jobs↔LinkedIn Job Posts tab switch within the Search page) must live in `AppContext`, not in a page/component's local `useState`. **This has already caused a real bug once** (search results and form inputs silently reset on every tab switch) — if you add new page-level state that should survive navigation, put it in a context hook, not local state.

All state hooks merge into one `AppContext` (`state/AppContext.tsx`), consumed via `useAppState()`.

## Conventions & Constraints

- **API keys stay server-side.** All Gemini/Apify calls route through `/api/*`. Never expose keys to the client bundle.
- **Graceful degradation is required.** Every AI path needs a local heuristic/rules-based fallback for API outages, timeouts, or missing credentials — see the pattern above.
- **No restrictive browser APIs.** Do not use `window.open`, `window.alert`, or similar for app-generated UI. Use in-app drawers, custom modals, toasts, and dynamic target transitions instead (iframe-safe). Plain `<a href target="_blank">` for viewing an external job posting is fine — that's normal navigation, not a popup.
- **Normalize before returning.** Never leak per-portal raw Apify schemas to the frontend — always go through `jobNormalize.ts`.
- **Job aggregation target**: ≥5 portals in under 10s where possible (real scraping often takes longer in practice; the 90s outer timeout in `routes/jobs.ts` reflects reality, not the original target).
- **Sidebar is a fixed, always-visible left rail** (`client/src/components/Sidebar.tsx`), not a toggleable hamburger drawer — there's no open/close state or backdrop anymore. Page content is offset via `paddingLeft: SIDEBAR_WIDTH_PX` in `App.tsx`. If you resize the sidebar, update `SIDEBAR_WIDTH_PX` (exported from `Sidebar.tsx`) rather than hardcoding the width in two places.

## Known Rough Edges (don't be surprised, don't silently "fix" without noticing the tradeoff)

- `parseYearsFromRange` (in `resumeFallback.ts`) computes experience duration from year-difference only, not months — a role spanning e.g. "Apr 2023 – Dec 2023" counts as 0 years. Minor undercount in aggregate experience totals; known, not yet fixed.
- The Guidance Layer's `gapType`/`path` derivation (`matchScoring.ts`) is a documented but genuinely heuristic mapping from the new rubric's dimension scores — the rubric itself has no such concept, so thresholds like "domain/seniority ≤1 is a hard filter" or "≥4 skill gaps is a structural mismatch" are tunable judgment calls, not something derived from the instructions doc. Treat them as adjustable, not sacred.
- `estimateDimensionsHeuristic()` (`matchScoring.ts`) — the batch-scoring fallback for when Gemini is unavailable/out of quota — is deliberately coarse (keyword overlap ratios, a small hardcoded domain-hint list, simple title-seniority regex). It's meant to keep list-view scores from going blank, not to rival Gemini's evidence-based scoring; results are marked `estimated: true` specifically so the UI doesn't present a guess as the real thing. Don't grow this back into the old ~300-line heuristic signal system — if it needs real improvement, that's a sign to fix the quota problem (caching, a higher-quota key) rather than the estimator.
- The resume fallback parser is tuned against real-world PDF extraction quirks (multi-line wrapped bullets, "Current" vs "Present" as an end-date, pipe-separated "Role | Company | Location" headers, PDF page-break artifacts like "-- 1 of 2 --", hyphens embedded in titles like "Product Manager - 2"). **If you touch `resumeFallback.ts`, test against an actual multi-page PDF resume via `pdf-parse`, not just clean synthetic text** — several real bugs here were only caught that way, not by unit-testing against tidy sample data.
- LinkedIn Job Posts' location filter is soft/best-effort text matching, not authoritative — the actor has no structured location field, so some genuinely-local posts will be excluded simply because they don't mention a location token in free text. Known, stated in the UI, not a bug to "fix" by trying to infer location more aggressively (that trades recall for precision in the wrong direction).
- LinkedIn Job Posts has no per-user usage cap on Apify or Gemini calls, and free-tier Apify accounts cap at 4 keywords/run — selecting more than 4 role titles means not all of them get searched in one run. Both are surfaced in the search panel's helper text rather than engineered around, same philosophy as the Gemini quota callout.

## When Editing

- Match existing file/module boundaries — resume, scraping, scoring, outreach, tracker are separate concerns in separate files.
- If adding a new job portal: extend the `Portal` union in both `types.ts` files, add an Apify actor mapping + input builder in `services/apify.ts`, and add a normalization adapter in `services/jobNormalize.ts`.
- If changing scoring: preserve `matchScoring.ts`'s single-source-of-truth pattern — `computeOverallScore()`/`bandFor()`/`classifyGapType()`/`selectPath()` must always run in deterministic TypeScript from the six dimension integers; never trust a final score or gapType/path directly from Gemini, only the 0-5 dimension values and (for the deep report) prose. Keep the two Gemini call shapes in sync if you change the rubric: `DEEP_REPORT_SCHEMA`/`buildDeepReportPrompt` (single job, full report) and `BATCH_SCORE_SCHEMA`/`buildBatchScorePrompt` (whole search, dimensions only, no prose) both need updating together, or list-view and deep-dive scores will silently diverge in meaning. Do not add a deterministic fallback for this route without discussing it first — the lack of one here is a deliberate, explicit exception to the Core Pattern above, not an oversight to "fix."
- If touching outreach: keep the five-part structure (Hook → Bridge → Ask → Honest Gap → Low-friction close, see `outreachFallback.ts` and the reference example embedded in the `routes/outreach.ts` prompt) and the direct/honest/low-pressure tone — avoid corporate-speak ("align with your goals", "translate into impact"). Update both `outreachFallback.ts` (deterministic) and the `routes/outreach.ts` prompt together — don't change one without the other. If touching regenerate specifically, keep the "genuinely different, not a reword" requirement on both the Gemini prompt and the fallback's variant rotation.
- If changing the LinkedIn Job Posts classification/extraction logic: verify against a real live trial run (small `max_posts`) before finalizing, not just the actor's documented schema — the real output has fields the README doesn't mention (`content.type === "job"` for structured job cards was only discovered this way). Keep the "never fabricate" bar high: prefer a generic placeholder (`"Hiring company"`) or omitting a field over inventing a specific value you're not confident in.
- Run `npm run typecheck -w server` and `-w client` after every change, and actually exercise the affected flow (curl the route, or drive it in a browser) before calling something fixed — this codebase has repeatedly had bugs that typechecked cleanly but were wrong at runtime (regex edge cases, timeout-vs-logic-failure confusion, state lifted to the wrong layer).
